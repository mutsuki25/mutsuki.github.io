### ソフトウェアライフサイクル（SLCP）の全体像
ソフトウェアには人間と同様に寿命があり、以下の3つのフェーズに大別される。
* **誕生（ニーズの発生）**：ビジネス上、またはシステム上の要求を具現化するために、企画や計画を立てる。ここで「どんなソフトウェアが必要か」「それによって業務がどう変わるか」を明確にする。
* **開発・運用**：設計、コーディング、テストを経て実稼働へと移り、その後の機能追加や性能改善、老朽化対策などの保守運用を行う。
* **廃止**：サービスの終了や、新規ソフトウェアへの全面的なリニューアルによって役目を終える。

### 要件定義の具体的な内容と意義
要件定義書は、プロジェクトに関わるすべての人間（顧客、営業、企画、設計者、ベンダ）にとっての「バイブル」であり、共通のスコープ（範囲）を認識するためのものである。

* **要件定義書に盛り込むべき項目**：
  システムの目的や概要、提供する機能だけでなく、システム構成図やソフトウェアブロック図、目標性能、他システムとのインターフェイス仕様、さらには運用面の注意事項、制限事項、将来的な拡張性、開発スケジュール、開発体制、最終的な納品物まで網羅的に決定する。漏れを防ぐためにテンプレート化する企業も多い。
* **実現性の評価（Feasibility Study）**：
  要件定義の段階で、そもそも実現可能かどうかを評価する「フィージビリティスタディ」を行う。これにより、機能の絞り込み、スケジュールの見直し、代替策の検討などを早期に行うことができる。
* **RFP（Request for Proposal：提案依頼書）との関係**：
  開発を外部委託する場合、この要件定義をベースにRFPを作成し、ベンダに対してシステム提案を要求する。

### 制作・調達における納品物の境界線
設計書に従ってシステムを構築する際、自社開発（内製）だけでなく、一部をパッケージ購入したり外部委託（調達）したりするケースがある。
外部委託の際は「ソースコードを納品物に含めるか否か」の合意が極めて重要。コードが含まれない場合、その後のメンテナンスや改造はすべてその委託先に縛られることになる。

### プロジェクトの2大特性
ソフトウェア開発に代表される「プロジェクト」と呼ばれる仕事には、必ず以下の2つの特徴がある。
* **有期性**：必ず始まりと終わり（期限）が決まっており、ダラダラと続くルーティンワークとは明確に区別される。
* **独自性**：これまでにない、独自の目的や成果物を達成するために組織される。

### バックキャスティングによる計画
プロジェクト業務を遂行する上では、今の立ち位置から順に進む「フォアキャスティング（インプット視点）」ではなく、あらかじめ定めたゴールから逆算して計画を立てる「バックキャスティング（アウトプット視点）」のアプローチが不可欠。
* **特徴**：目標（イシュー）を具体的かつ達成価値のあるものとして明確に設定し、そこに至る筋道や目標自体を、状況に応じて適宜修正しながら進めていく。

### 開発プロセスの分類と特徴
開発を進める枠組みには、歴史的な変遷とそれぞれのメリット・デメリットがある。

* **ウォーターフォールモデル（従来型）**：
  要件定義、設計、実装、テストといった工程を、水が上から下に流れるように順番に進める。原則として前の工程に戻らないため計画は立てやすいが、終盤のテスト段階でミスが発覚した際の手戻りリスクが非常に大きい。
* **プロトタイプモデル**：
  開発の初期段階で試作品（プロトタイプ）を素早く作り、ユーザーに確認してもらう手法。認識のズレを早い段階で解消できるが、試作の繰り返しによるコストやスケジュールの管理が難しくなりがち。
* **スパイラルモデル**：
  システムをいくつかのコンポーネント（サブシステム）に分割し、それぞれに対して設計、試作、評価のサイクルを繰り返しながら、段階的に全体の完成度を高めていく手法。
* **アジャイルプロセスとスクラム**：
  変化の激しい現代において、要求の変更を歓迎する柔軟な開発スタイル。数週間〜数ヶ月という短いサイクル（イテレーション）で動作するソフトウェアの納品を繰り返し、顧客満足度を最優先する。ビジネス側と開発者が日々協力し、顔を突き合わせて情報伝達を行うのが特徴。XP（eXtreme Programming）ではテスト駆動型開発やペアプログラミングなどが実践され、タスク管理には「スクラム」や「カンバン」といった手法が用いられる。

### WBS（Work Breakdown Structure）の本質とメリット
WBSとは、プロジェクトの目標達成と必要な要素成果物の生成のために、チームが実行する作業を「要素成果物」を主体に階層的に要素分解した構成図のこと。
大きなプロジェクトを細かな作業（Work）に分解（Breakdown）し、作業順に並べることで、以下の強力なメリットが生まれる。
* **スコープの明確化**：何を行うべきかがクリアになり、同時に「やらないこと」も明確になる。
* **作業の洗い出し**：直感に頼らず、やるべき作業を漏れなくリストアップできる。
* **見積もりと計画の具体化**：タスクが細分化されるため、必要な期間や工数の予測が立ちやすくなり、プロジェクト実施時はこれに則って動くだけの状態を作れる。

### WBS作成の4つのステップ
WBSは、以下の手順に沿って厳密に作成していく必要がある。
1. **スコープを明確にする**：ボトムアップ（積み上げ）ではなく、バックキャスティングによる「トップダウンアプローチ」で全体の枠組みを決める。
2. **大きな作業をグルーピング（フェーズ化）する**：例えば開発なら「データ収集」「データ分析」「可視化」といった大枠、イベントなら「準備」「設営」「本番」「片付け」などのフェーズに分ける。
3. **作業の相互関連（依存関係）を考える**：「データ収集をしてから分析する」のように、作業の前後関係や「これが終わらないと次が進まない」というつながりを整理する。
4. **各グループの作業を徹底的に洗い出す**：この際、抜け漏れや重複がない状態、すなわち **MECE（Mutually Exclusive and Collectively Exhaustive）** を意識して最小単位のタスクまでブレイクダウンする。

### コーディングとスタイルガイド（PEP8 / flake8）
* **目的**: 読みやすさ（可読性）の向上とプロジェクト内での一貫性の保持
* **インデント**: 1レベルごとにスペース4つ（タブは使用しない）
* **行の長さ**: 通常のコードは79文字以内、コメント等は72文字以内
* **インポート順**: 1.標準ライブラリ ➔ 2.サードパーティ ➔ 3.ローカルライブラリ（各アルファベット順）
* **命名規則**: クラス名はCapWords（CamelCase）、関数・変数・メソッド名は小文字＋アンダースコア
* **禁止事項**: 見分けがつきにくい `l`, `O`, `I` の1文字変数名、1行での複数インポートや複数文の記述
* **ツール**: 静的解析ツール **flake8** による規約違反（E/Wコード）やコードエラー（Fコード）の自動チェック

### Git/GitHubを用いたチーム開発
* **GitHub機能**: タスク管理を行う **Issues** や **Projects**、クラウド開発環境の **Codespaces**

### CI/CD（継続的インテグレーション／継続的デリバリー）
* **CI（継続的インテグレーション）**: コード変更を頻繁に統合し、ビルド・テストを自動実行してバグを早期発見するプロセス
* **CD（継続的デリバリー）**: テスト済みのコードを自動でテスト/本番環境へデプロイし、リリースを迅速化するプロセス
* **GitHub Actions**: `.github/workflows/` 内の **YAML（.yml）ファイル** で自動化を設定
* **YAMLの構成要素**: トリガーを定義する `on`、実行環境を指定する `jobs`、手順を定義する `steps`

### GitHub PagesとJekyllを用いたWebサイト公開
* **概要**: リモートリポジトリのMarkdownファイルを、静的サイトジェネレーター **Jekyll** でHTML化して無料公開する機能
